# Repository Guidelines

## Project Structure & Module Organization

This repository contains firmware for a three-controller targeting car. `附件二：源码/M0车源码` is the MSPM0G3507 vehicle controller (motors, encoders, and line-following state machine). `附件二：源码/CCDM0解算` is a second MSPM0G3507 project that samples the 128-pixel line-scan sensor and sends line center/width data. `附件二：源码/云台main源码/TIcup_AutoTrackerCar-main` is the STM32H750 gimbal controller, including vision/IMU UART handling, stepper motion, laser control, and the CubeMX configuration.

Keep application logic in the existing source folders (`Core/Src`, `User`, `keil/motor`); treat `build/`, `MDK-ARM/*/*.o`, `Objects/`, `.axf`, and `.hex` as generated artifacts. The `知识库` folder is reference material, not firmware input.

## Build, Test, and Development Commands

Use the matching IDE/project for each target:

```powershell
UV4.exe -b "附件二：源码\M0车源码\keil\empty_LP_MSPM0G3507_nortos_keil.uvprojx"
UV4.exe -b "附件二：源码\CCDM0解算\keil\adc12_single_conversion_LP_MSPM0G3507_nortos_keil.uvprojx"
cd "附件二：源码\云台main源码\TIcup_AutoTrackerCar-main"
cmake --preset Debug; cmake --build --preset Debug
```

The MSPM0 projects require `mspm0_sdk@2.04.00.06`; preserve the `.syscfg` pin assignments unless the hardware changes. The H750 project also opens in `MDK-ARM/TMC2209.uvprojx` and targets STM32CubeH7 1.12.1.

## Coding Style & Naming Conventions

Follow the surrounding C style: four-space indentation in H750 code and tabs in legacy MSPM0 files; do not reformat unrelated files. Use `snake_case` for functions and variables, `UPPER_CASE` for macros, and retain peripheral names generated by SysConfig/CubeMX. Put board-specific constants near existing calibration values and document units (pixels, encoder counts, microseconds).

## Testing Guidelines

There is no host-side test framework. Verify changes in stages: CCD serial output, motor direction/encoder sign, stationary gimbal aiming, then full vehicle motion. Record the board, firmware revision, supply voltage, and calibration inputs. Never test the laser without a controlled target and eye-safe setup.

## Commit & Pull Request Guidelines

The repository has no commit history yet. Use concise imperative commits, for example `fix: clamp gimbal scan range`. Keep each commit focused. PRs should state affected controller(s), hardware assumptions, build result, bench-test evidence, and any changed pin, UART, or calibration values. Include photos or serial logs for behavior changes.
